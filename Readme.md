# RequestService

## Описание
Сервис для автоматизации процесса запроса и обработки бухгалтерских справок

## Архитектура

- **Domain**: сущности, перечисления, исключения, контракты репозиториев.
- **Application**: DTO, сервисный слой, бизнес-правила, валидация переходов статусов.
- **Infrastructure**: EF Core, PostgreSQL, реализация репозиториев, DI-конфигурация.
- **API**: HTTP-контроллеры, middleware обработки ошибок, Swagger.

**Стек**: .NET 9, C# 13, ASP.NET Core, EF Core, PostgreSQL, Docker.

## Запуск 

1. Поднятие БД на PostgreSQL в docker контейнере
```bash
docker compose up -d --build
```
2. Применение миграции:
```
dotnet ef database update --project RequestService.Infrastructure --startup-project RequestService.API
```

## API

```https://localhost:7208/swagger/index.html```

| Метод  | Endpoint                                | Описание                       |
|--------|-----------------------------------------|--------------------------------|
| POST   | `/api/documentrequests`                 | Создать заявку на справку      |
| GET    | `/api/documentrequests/employee/{name}` | Список заявок сотрудника       |
| GET    | `/api/documentrequests/{id}`            | Детали конкретной заявки       |
| PATCH  | `/api/documentrequests/{id}/status`     | Изменить статус                |


## Вопросы рефлексии

### 1. Какое архитектурное решение вы считаете самым рискованным и почему?
**Изменение статусов без жёстких ограничений на уровне БД.**
При высокой нагруженности два бухгалтера могут одновременно забрать заявку со статусом `InProgress` и обновить его на `Ready`/`Rejected`, минуя проверку `IsValidTransition`.

### 2. Где вы сознательно оставили технический долг? Как бы его закрыли при наличии времени?
- Валидация через атрибуты в DTO/сущностях. Заменил бы на `FluentAPI`. Это даст переиспользуемость и отвяжет валидацию от модели данных.
- Отсутствие тестов. Добавил бы тесты.
- В исключении добавил бы перечисление для типов ошибок.

### 3. Что бы вы изменили, если бы знали, что сервис станет высоконагруженным?
- **Асинхронная обработка заявок**: добавление брокера сообщений (RabbitMQ/Kafka). Заявка падает в очередь, сервис возвращает, что получил заявку, фоновый процесс обрабатывает и отправляет статус заявки.
- **Кэширование**: использовать Redis для хранение данных частых запросов.
- **Разделенять на чтение и изменение данных**: использовать CQRS-паттерн.
